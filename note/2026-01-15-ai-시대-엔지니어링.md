# AI 시대, 개발자와 엔지니어의 경계

**태그**: `AI`, `엔지니어링`, `개발 방법론`, `회고`

## 핵심 요약

> 코드는 AI에게 맡기고, 판(방향과 책임)은 인간이 가져간다.

## AI vs 인간의 역할 구분

| 영역 | AI가 잘하는 것 | 인간이 해야 하는 것 |
|------|---------------|-------------------|
| 패턴 | 검증된 패턴을 빠르게 적용 | 낯선 선택, 기존 방식을 버리는 결정 |
| 정보 | 문서/스펙을 읽고 구현 | 문서에 없는 상황(조직, 고객, 우선순위) 파악 |
| 최적화 | 주어진 조건 내 최적해 | 조건 자체를 바꿀지 결정 |
| 지식 | How (문법, 설정, 옵션) | Why (도메인, 운영 기준, 우선순위) |

## 개발자 vs 엔지니어

- **개발자**: 요구사항 → 작동하는 코드 (전술 실행)
- **엔지니어**: 시스템 구조 설계 + 운영까지 책임 (판 짜기)

### 엔지니어가 답해야 할 질문들
- 이 기능은 어디에 붙여야 하는가? (경계)
- 데이터는 누가 소유하고 변경 승인은? (책임)
- 트래픽 증가 시 어디가 먼저 터지는가? (병목)
- 장애 시 복구 절차는? (운영)
- 빠르게 vs 안전하게? (트레이드오프)

## 개발 계획서 먼저 쓰기

코드 전에 **전략 결정 문서**를 작성한다.

```markdown
## 목표
- 이번 변경으로 무엇이 좋아져야 하는가?

## 제약/전제
- 일정, 정책, 레거시, 운영 인력, 외부 의존성

## 전략 후보
- A안: 빠르지만 리스크 큼
- B안: 느리지만 운영 쉬움
- C안: 이번엔 포기

## 선택 기준
- 우선순위 (비용/안정성/속도/정합성)

## 운영/복구
- 관측 포인트, 롤백 가능 여부, 복구 절차

## 작업 범위
- 하는 것 / 하지 않는 것
```

**코드 리뷰보다 계획서 리뷰가 더 큰 비용을 절약한다.**

## 대체될 수 없는 엔지니어의 조건

1. **불확실성 속 선택**: 정답 없는 상황에서 책임지는 결정
2. **현장 감각**: 문서에 없는 조직/고객 맥락 파악
3. **조건 재정의**: "이 구조로는 못 합니다" 라고 말할 수 있는 용기
4. **우선순위 합의**: 팀을 하나의 기준으로 묶는 능력
5. **운영 책임**: 장애 시 복구까지 닫는 것

## 결론

- 구현 속도는 AI가 해결
- 방향과 책임은 인간의 영역으로 남음
- 코드는 공용화되고, 판은 더 비싸진다
